# Lessons Learned

Append-only log of non-obvious decisions and the incidents that caused them.
New entries go at the top of the relevant section. Format: one-line rule → short reason.

## Scoring schema evolution
- **Digest explanations must be deterministic, not LLM-generated** — R4-02 generates "Why Apply / Watch-outs / What's holding this back / Still worth a look" from sub-score + aggregate output only. User rule 2026-04-23: an LLM narrative on top of interpretable sub-scores adds variance and undermines trust — "I want to see exactly why the system recommended this, without second-guessing a generated paragraph." Implementation: `src/digest/explain.py` — pure functions over row dict, no I/O, no LLM. Bullet dedup preserves first occurrence so the same top_reason doesn't appear twice. Fallback: when aggregate didn't surface top_reasons/top_risks (rare edge case), derive bullets from strong/weak sub-scores directly.
- **Pending-feedback diff needs a snapshot, not a count** — to remind the user about specific cards from the previous digest that have no verdict yet, R4-02 persists `output/last_digest.json` with title + company + url + recommendation + final_score for every Apply / Review / borderline-Skip row shown. Without the snapshot we can only say "you have 5 unreviewed", which doesn't help the user find the specific cards. Snapshot is overwritten on every send so it's always exactly "last digest", never "some past digest". Requires no DB migration.
- **Theme clustering works best with a small fixed taxonomy, not free-text LLM summarisation** — R4-02 clusters Reject verdicts into ~9 themes via regex on `user_reason + title`. Benefits: deterministic (same reject → same bucket on every run), explainable (user can see why "too technical" and "heavy simulation work" land in the same bucket), zero quota cost. First-match-wins ordering matters — specific themes (Too software / SaaS-focused) must come before generic ones (Wrong industry) so they win when both match. Unmatched reasons fall into "Other reason" so totals still sum to reject count.
- **`industry_proximity` classifier column vs `score_industry_proximity` sub-score column: rename rather than shadow** — R3-05 lesson reapplied in R4-02 render: when the digest pulls from a DataFrame carrying BOTH a 0-1 classifier proximity AND a 0-10 sub-score, the sub-score column is prefixed `score_` and explicitly listed in `_SUBSCORES`. No "pick whichever is present" logic — the renderer always looks at `score_industry_proximity`. Defensive default is 0 when missing.
- **Gate heavy LLM on sub-scorer verdict, not just pre-rank cap** — R3-05 pipeline wires 6 sub-scorers into main.py AFTER pre-rank and BEFORE the heavy matcher LLM. Matcher now only fires on `recommendation in ("Apply","Review")`; auto-Skip rows bypass entirely. User rule 2026-04-23: "LLM should evaluate only meaningful surviving jobs, not waste tokens on obviously low-priority or already-rejected jobs." Cheap skill-extract LLM runs one step earlier, disk-cached by URL + JD-hash. Observability line prints skill_llm_calls / cache_hits / extraction_failures per run so we can see where LLM budget actually goes. Bypassed rows still land in Sheets + digest (match_score mirrored from final_score, Review → Maybe for legacy consumers) so no visibility is lost.
- **Sub-score pipeline must attach audit columns from day one, not retrofit later** — R3-05 attaches drop_stage, drop_reason, skill_extract_source, scored_by_llm, used_skill_cache, llm_stage_skipped_reason on the same pass that attaches score columns. Rationale: the audit columns cost ~20 extra lines during orchestration but retrofitting "which rows hit the LLM cache" after the fact requires re-reading on-disk caches keyed by dead URLs. Design principle: any stage that decides a row's fate must record WHY on the row itself.
- **`industry_proximity` column name collision is easy to miss** — R3 industry classifier attaches `industry_proximity` (0-1 float) to the DataFrame; R3-05 sub-scorer outputs a 0-10 integer. Using the same column name would have silently overwritten the classifier value and broken the aggregate's demote logic. Fix: sub-score column prefix `score_` (e.g. `score_industry_proximity`) — clear separation from classifier output. Rule: when composing stages, prefix score columns so they can't collide with classifier columns at the same semantic layer.
- **Canonical-name rename is a clearer fix than synonym enrichment when the name itself is wrong** — R3-02b started by trying to add synonyms to "Mechanical engineering fundamentals" to catch mechanical-engineering JDs. User pushed back 2026-04-22: the canonical is itself too broad — it matches component-level / material / simulation jobs Ava does NOT want. Fix was renaming to "Product development for physical products" and narrowing synonyms to product-level phrasings. Rule: if a canonical name frames the skill too broadly, enriching its synonyms makes the problem worse; rename + narrow first, then enrich.
- **Tool-mention ≠ expertise; separate into adjacent-Low vs forbidden-depth** — LLM extractions often surface bare tool names (ANSYS, FEA, FMEA, SolidWorks simulation) even when the role isn't simulation-focused. Naive YAML would put these all in ONE bucket — either forbidden (wrong — Ava has tool familiarity) or verified (wrong — not her strength). Solution: two parallel entries — adjacent "Design validation tools familiarity" (Low strength) catches bare-tool mentions (+0.6 pts), while forbidden "Mechanical simulation / FEA / CFD depth" only fires on depth-qualified framings (fea primary / fea analyst / simulation engineer / advanced fea / structural simulation expertise). A pure simulation role still hard-rejects; a product-development role that happens to use ANSYS is appropriately lifted a tiny bit.
- **Avoid double-counting across canonicals when identity and activity both match** — R3-02b initial draft put "prototype development and validation" under BOTH the renamed Product development entry AND the existing Prototyping entry. User caught this 2026-04-22: identity (physical product development) should be a separate canonical from activity (prototyping). Same JD phrase matching two verified canonicals double-counts points. Rule: each JD skill phrase should map to exactly one primary canonical; the other canonicals should sit at a different conceptual layer.
- **Aggregate guardrails must be independent, not additive, when one dimension is structurally weak** — R3-04 introduced a hard_skill ≤ 8/30 guardrail that demotes to Review even if total score ≥ 70. User rule: a low hard-skill sub-score must not be masked by other strong sub-scores. This is an "independent demoter," not a penalty — it fires alone. Rationale: when hard_skill coverage is known-temporary (pending R3-02b), threshold recalibration would create churn later; a deterministic guardrail is simpler to reason about and trivially removed once coverage improves.
- **Apply/Review/Skip must be deterministic post-scoring, not LLM-driven** — Aggregate routing is a pure function: Skip (blockers or score<50) > Review (band 50-69 OR any demoter) > Apply (score≥70 + clean). User-locked demoters: seniority_mismatch_flag, role_rescued, 2+ forbidden_optional, 1 forbidden_optional + weak hard_skill, hard_skill ≤ 8. industry=unknown alone is informational only. Benefit: no LLM variance in tier assignment; easy to test boundaries deterministically.
- **LLM skill extraction is verbatim and hyper-specific; YAML synonyms must be broad enough to catch it** — R3-02 dry-run on TE Connectivity Senior R&D PDE: Gemini extracted 38 accurate phrases ("FEM simulations", "injection-molded components", "precision-stamped electro-mechanical components") but only 1 matched skills.yaml (via "creo"). Lesson: LLMs surface the JD's own vocabulary, not the canonical taxonomy vocabulary. Fix options in descending effort: fuzzy/embedding match (heavy), dual-field extraction (literal + category, prompt tweak), or synonym enrichment based on observed cache (fastest). Decision for V1: ship the scorer (logic is correct) and defer taxonomy work to R3-02b — a vocabulary task, not a scoring-logic task.
- **Cache key must include a JD-hash, not just URL** — same job_url can carry an edited JD body days later (employer updates must-haves, adds sponsorship clause). URL-only cache would return stale extractions. Lesson from R3-02 design (user-mandated 2026-04-22): compose cache key as `{url}|{sha256(normalized_jd)[:16]}`; normalize whitespace before hashing so trivial reformatting doesn't bust cache unnecessarily.
- **Separate the extraction chain from the scoring chain** — R3-02 uses a minimal 2-provider chain (Gemini primary → Groq fallback) while full matcher.py keeps the 5-provider chain. Reason: extraction is a smaller, cheaper, more predictable task — no need to pay the cost of the reasoning-grade providers. `call_llm(config=filtered_config)` pattern works well: reuse the router code, restrict the model list.
- **Weak vs strong deliverable keywords need different weights, not just inclusion** — R3-01 started with `hands-on`, `player-coach`, `0→1` at the same +1 weight as `research`/`prototype`/`validate`. User feedback 2026-04-22: "too vague and too easy to trigger in jobs that are not actually a fit." Fix: split into two lists — strong (+1) for grounded verbs (research, prototype, validate, benchmark, requirements, feature definition, insight synthesis, etc.) and weak (+0.5) for fuzzy startup language. Same bonus cap (+5). Rule: a keyword's weight must match how reliably it predicts real fit, not how often it sounds appealing.
- **Deliverable keywords must reflect real past work, not aspirational startup vocabulary** — initial R3-01 list leaned on "player-coach", "0→1", "hands-on" (buzzwords). User pushed back 2026-04-22 with the actual terms from her work: `benchmark`, `requirements`, `feature definition`, `solution evaluation`, `insight synthesis`. Rule for personal-fit scoring: ground keywords in the candidate's real deliverables, not the ecosystem's buzzwords.
- **"Alignment" words are false negatives when used as blanket anti-patterns** — `cross-functional alignment` and `stakeholder alignment` appear in perfectly good jobs (including the user's own past roles). Removing them from the anti-pattern list fixed a systematic 1-3pt drag on desired-bucket roles. Kept as anti-patterns: `ppt`, `slide deck`, `waterfall`, `steering committee` (unambiguously coordination-/process-heavy). `roadmap alignment` kept but ONLY fires when a hard anti-pattern also fires (combo rule).
- **Safety-net zeroing must beat bonuses, not just base** — first R3-01 draft had `final = max(0, base + bonus − penalty)` where `base=0` for hard-reject excluded; 5 deliverable hits lifted the final to 5. Fix: explicit `if bucket == "excluded" and would_hard_reject: final = 0`, short-circuiting bonuses. Rule: if upstream stage considers a job rejectable, no downstream scorer should be able to partially undo that with pattern matches.
- **Seniority classifier must separate hard-reject from mismatch-flag** — R2-03 dry-run showed 19 clear early-career jobs (all intern/trainee/graduate-program) alongside 43 title-based junior and 7 head-of roles. User rule (2026-04-22): hard-reject is ONLY for the 19; junior/head-of/mid-level must be flagged + routed to Review (never dropped). Implementation: `src/classify/seniority.py` returns `seniority_hard_reject` (true only for explicit early-career) and `seniority_mismatch_flag` (true for junior/head-of/mid); Stage 3b drops the former, keeps the latter. Soft penalty values are deferred to R3 so we don't double-penalize.
- **Word-boundary regex catches obvious seniority false positives too** — naive substring matching on `intern` matched "international" product managers; `trainee` matched "trainer platform" roles. Fix mirrors industries/skills: all seniority detectors use `(?i)\b…\b`. Dedicated tests: `test_word_boundary_no_false_positive_on_intern`, `test_word_boundary_no_false_positive_on_trainee`.
- **Stage 3b print/log sites need a `gate` discriminator when multiple gates share the same rejection list** — when R2-03 added seniority hard-rejects alongside the existing role-family gate, the print loop broke on missing `role_family`/`role_confidence` keys. Fix: every rejection record carries `gate: "role_family" | "seniority"`; downstream renderers branch on that tag. Lesson generalizes: if you merge two reject streams into one list, tag the source.
- **Ambiguous role-families MUST require a title match** — first R2-01 dry-run had 35% of jobs tagged `ambiguous` (e.g. 139 strategy_consultant hits) because JD phrases in `favors_good_fit` (innovation, design thinking, prototype, hands-on) accumulated to 0.4+ score without any title hit. Dropped to 3% after requiring `title_hits` in `_score_candidate`. Rule: ambiguous families exist to disambiguate a SPECIFIC title pattern; JD-only matches are not classifications, they're noise.
- **Rescue-and-promote beats rescue-only** — when an excluded-title job has rescue signals AND a desired/ambiguous alternative scores higher, PROMOTE the alternative to primary (not just secondary). Locked per user 2026-04-22: "rescued but still only tagged as excluded is not enough — I want rescued jobs to be usable, not just spared from rejection."
- **Skill labels must describe ACTUAL scope, not superset** — broad labels like "Product vision & roadmap", "GTM strategy", "Technical PRD", "UX design fundamentals" encourage scoring systems to match Ava to roles she's not actually aligned with. Fix: split / narrow / rename with scope clarity. Examples from skills.yaml: "Product vision & roadmap" split into "Product strategy input" (verified) + "Product vision" (adjacent); "GTM strategy" narrowed to "Product positioning and GTM input"; "Technical PRD definition" narrowed to "Feature requirements definition"; "UX design fundamentals" removed (implied UI/visual craft) and replaced with "Research synthesis". Rule: if the canonical name reads like the full senior version of the skill, bucket may over-score.
- **"Junior" is not a blanket hard-reject** — title like "Junior Product Manager" at some companies means pay grade, not entry-level. User rule 2026-04-20: hard-reject only unambiguous intern/graduate/trainee/stagiair; "junior" in title → soft penalty + seniority_mismatch flag → Review.
- **Constraint phrase triggers need scope awareness** — "entry level" or "0-2 years" appearing anywhere in a JD blob does NOT mean the role is entry-level (may describe a product tier, team spec, or historical note). First dry-run of `constraints.yaml` hard-rejected 116 jobs for "seniority_junior_role" on this kind of false positive. Fix: two-channel trigger in `constraints.yaml` — `title_patterns` scans the title FIELD ONLY; `phrases_any_in_role_context` requires explicit role-defining phrases ("graduate program", "internship program"); avoid generic seniority keywords in broad `phrases_any`.
- **Taxonomy hints must use word-boundary regex, never substring matching** — initial dry-run of `industries.yaml` matched bare `ev` as substring → "development"/"every" false-triggered automotive → 92% of jobs misclassified. Fix: classifier uses `re.compile(r'\b(hint1|hint2|...)\b', re.I)` on both `company_hints` and `jd_hints`. Also: avoid 2-letter tokens in hints (e.g. bare `ev`, `cv`); use phrases like `electric vehicle` / `ev/phev` / `computer vision`. Same rule applied to `skills.yaml` synonyms.
- **Industry classification is advisory, never a hard-reject** — JD-based industry matching catches incidental mentions (e.g. a ship-building JD mentioning "ERP" gets classified as enterprise_saas). Since industry proximity only affects 0-10 points in the final score and NEVER rejects (per user rule), this noise is bounded. Company-name match is authoritative; JD match is fallback.
- **Certification ≠ verified skill** — holding a certificate or a supplementary project doesn't make a skill a core strength. ML/CV was initially placed in `verified` (CUHK ML certificate + applied CV in robotics), but dry-run showed it pulling in ML-engineer postings as apparent fits. Correct bucket: `adjacent`, with `evidence_strength: Low`. Rule: if the skill isn't something the user would confidently claim on a CV as a primary strength, it's adjacent at best.
- **Synonym "cv" collides with "curriculum vitae"** — generic 2-letter initialisms in `synonyms` lists produce false positives across unrelated JD sections. Prefer verbose synonyms ("computer vision") over shorthand.
- **Title regex alone must never hard-reject** — early role-family classifiers that rejected on title keywords produced false negatives (e.g. "Electrical Architect: Mechatronics & Medical Innovation" at Philips was a good fit). Design rule in `config/personal_fit/roles.yaml`: excluded_families have `jd_signals_rescuing` + `reject_confidence_threshold`, and `classifier_defaults.review_on_conflict=true`. Excluded title + any rescue signal → Review, not Skip.
- **Score-only ranking hid work-style risk** — a single 0-100 MatchScore can't express that a decent-on-paper PM in a sales-led matrix org is worse than a lean hands-on innovation role. 4-block V1 redesign replaces it with 7 dimensions (Research/Innovation/Hands-on/Strategy higher-better, AlignmentBurden/ProcessHeaviness lower-better shown as "risk", TeamStyleFit) + AntiPatternRisk + deterministic SuggestedAction (6 labels). Score is still primary ranking signal; dimensions drive decision quality.
- **SuggestedAction must be computed post-LLM, not by the LLM** — LLMs are inconsistent at mapping score+risk thresholds to action labels. `_compute_suggested_action()` in matcher.py is deterministic: blocker → Skip, head-of regex → Seniority mismatch, (score, risk_level) → one of Strong Apply/Apply/Review/Skip/Good role risky org. Overrides whatever the LLM returned.
- **Backward-compat for schema changes: render both old and new in parallel** — ~80 existing Sheet rows have old WhyFit/Gaps schema; only newly scored jobs get GreenFlags/RedFlags. `_parse_card` extracts both; template prefers new, falls back to legacy. No rescoring campaign needed.

## Scraping
- **JobSpy LinkedIn ignores `location` string for non-US countries** — passing `location="Netherlands"` returns ~70% global results because JobSpy only sends the free-text location, not LinkedIn's `geoId`. Fix: monkey-patch `jobspy.linkedin.LinkedIn.__init__` to wrap `session.get` so every request to `/jobs-guest/jobs/api/seeMoreJobPostings/search` injects `geoId=102890719` (NL). Raised NL hit rate from ~30% to 100%. Implementation: `src/jobspy_patches.py` — imported at top of `src/scraper.py` before any `scrape_jobs` call.
- **LinkedIn via JobSpy always returns empty descriptions** — trafilatura fallback is mandatory, not optional.
- **Indeed / Google Jobs hang on Windows** — fixed with per-platform `concurrent.futures.ThreadPoolExecutor` + 90s timeout; hung thread is abandoned (not killed) but does not block remaining platforms. Glassdoor left disabled (API errors).
- **Glassdoor API is unstable** — keep disabled until upstream is fixed.
- **JobSpy location filter is unreliable** — `location="Netherlands"` in the scrape call does not prevent LinkedIn from returning global results. Must post-filter by the job's `location` field after scraping (Step 2.5). Pattern: keep if location ends with ", NL", == "NL", contains "Netherlands/Nederland", is empty, or is_remote=True.
- **Empty location is ambiguous, not a safe default** — jobs with empty location can be anywhere (e.g. "AVL in Türkiye" had empty location but is clearly Turkey). Current policy: keep empty locations (can't hard-reject without knowing), rely on LLM to score these appropriately. Acceptable edge case.

## Google Sheets
- **Adding columns to HEADERS after Sheet creation causes column shift** — the existing sheet header row doesn't update automatically; all new columns write to the wrong position. Symptom: "Job URL" shows "Unknown" because the URL is written 1+ columns to the right of the "Job URL" header. Fix: `_repair_jobs_headers()` in `setup_tabs()` compares live headers to HEADERS and overwrites row 1 if mismatched. Call `setup_tabs()` at pipeline start to auto-repair.
- **seen_jobs.json dedup blocks --resend re-scoring** — `score_all_jobs` loads seen_jobs.json and skips already-scored URLs. For re-score use cases (--resend, testing), pass `force=True` to bypass. Don't clear seen_jobs.json as it protects against expensive re-scoring in normal runs.

## Telegram bot
- **Verdict URL collision** — if Sheet's "Job URL" column is empty or contains "Unknown", `record_verdict()` keys all verdicts on that same string. Every new verdict supersedes all prior ones, leaving only 1 live entry. Fix: construct fallback URL from numeric LinkedIn job_id (`/jobs/view/{id}`) extracted from email deeplink before storing.
- **Bot conflict on quick restart** — Telegram holds the getUpdates long-poll connection for ~30s after a bot dies. Starting a new instance immediately gets "Conflict: terminated by other getUpdates request". Fix: call `getUpdates?timeout=1` once at startup to take over the slot before launching the polling loop.
- **`python src/tg_bot.py &` in Claude's bash tool does not persist** — the subprocess is killed when the parent bash process exits. Bot must be run from the user's own terminal, not via a background bash task.
- **Telegram `?start=` rejects full URLs** — only `[A-Za-z0-9_-]`, max 64 chars allowed. Indeed/Google job URLs URL-encoded as the deeplink payload broke all non-LinkedIn Skip/Good buttons. Fix: `_job_id_from_url` in notifier.py extracts short IDs — LinkedIn numeric ID, `in_<jk>` for Indeed, `hx_<md5[:12]>` for others. `_find_job_row` strips the 3-char prefix before searching the Sheet.
- **--resend jobs not in Sheet → deeplink shows "Unknown company"** — `--resend` skips Sheet write so bot can't find job metadata. Fix: `_lookup_job_from_cache(job_id)` falls back to `enriched_jobs.pkl` using the jk/numeric needle as substring match on job_url. Every job ID must always be resolvable — Sheet first, cache second.
- **LinkedIn returns only ~17 NL jobs per keyword** — JobSpy `location="Netherlands"` is unreliable for LinkedIn, returns global results. With results_per_keyword=20 and 198/261 non-NL filtered, only 63 NL LinkedIn jobs survive. Fix: raise results_per_keyword to 50 to increase NL coverage.
- **Skip session expiry on bot restart** — `pending_skip` state lives in `context.user_data` (in-memory); typed note silently dropped if bot restarts between "Add note" and user typing. Fix: persist to `output/pending_skip.json`, restore in `handle_text_message` when in-memory state is missing.

## Language detection
- **langdetect is non-deterministic** — replaced with langid for reliable Dutch detection.
- **langid log-prob is NOT a confidence score** — it scales with text length (longer text → more negative). Never threshold on it; just check `top language == "nl"`. A prior `confidence > -100` threshold silently disabled Dutch JD detection for full-length descriptions.

## LLM
- **Gemini 2.5 Flash thinking tokens consume max_output_tokens** — must set `thinking_budget=0` for JSON generation, otherwise output is truncated at ~200 chars.
- **5 providers configured with auto-fallback** — Gemini + Groq (priority 1), NVIDIA NIM + Mistral (priority 2), OpenRouter (priority 3). Config in `config/llm_config.yaml`. Provider type `openai_compat` works for any OpenAI-compatible API via custom `base_url`.
- **OpenRouter free tier rate-limits aggressively** — 15s cooldown between calls, multi-model rotation required, ~3 min per job at worst.

## Data quality
- **Salary metadata from JobSpy can be garbage** — validate min > 100, swap if inverted, reject non-numeric.
- **IND sponsor register is an HTML table** — parse with trafilatura, not as CSV.
- **Company names in IND register include B.V./N.V.** — rapidfuzz fuzzy match with suffix stripping needed.

## Runtime / infra
- **Python stdout buffering on Windows** — run with `PYTHONUNBUFFERED=1` and `-u` for background tasks.
- **Windows cp1252 crashes on Unicode arrows / CJK output** — reconfigure stdout/stderr to UTF-8 at the top of any entry point.
- **OpenBLAS thread contention can OOM pandas** — set `OPENBLAS_NUM_THREADS=1 MKL_NUM_THREADS=1 OMP_NUM_THREADS=1` before importing numpy/pandas in constrained environments.

## Deprecated decisions (kept for context)
- **Phase 2 CV tailoring was removed 2026-04-14** — generator, Flask review UI, PDF templates all deleted. LLM-generated CVs could not hit the quality bar (fabricated skills, wrong experience order, no photo, poor layout) and maintenance cost outweighed value. Focus is now daily scoring + fast feedback loop only.
- **5-sample feedback threshold (old rule)** — replaced with 1-sample nudge + pattern detection in the new design.
